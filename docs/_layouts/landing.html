Introduction to Secretless Code

Secretless is a connection broker which relieves client applications of the need to directly handle secrets to backend services such as databases, web services, SSH connections, or any other TCP-based service.

Why Secretless?

“Secretless” is designed to solve two problems. The first is loss or theft of credentials, which can occur by:

  1. Accidental leakage e.g. checked in plaintext to source control
  2. An attack on a privileged user, e.g. phishing, followed by privilege escalation
  3. A vulnerability in an application (e.g. remote code execution), followed by credential theft

The second is downtime which is caused when applications or services do not respond to credential rotation, and crash or get locked out of backend data sources as a result

Keeping secrets in a vault is a good practice. However, when a client “checks out” a secret from a vault, where does it go? Often, directly into unprotected application memory; the client which has “checked out” the secret from the vault is now part of the threat surface. There are a stringent set of best practices and recommendations that should be be followed by every user and application which checks secrets out of a vault. But this is a very tall order, because the list of best practices is quite long and technical.

It’s a losing battle to try and train every user and application developer about how to safely handle a secret once they’ve obtained it. There are so many things that can go wrong, and people (and applications) have better things to worry about than babysitting secrets (such doing their jobs).

“Secretless” alleviates users and applications from the burden of handling secrets. Instead of checking out secrets, when a user or application needs access to something important, it uses Secretless to establish the connection on its behalf.

Secretless is ONE piece of software is specifically written to be really good at interacting with all kinds of different vaults. It does a really good job of taking care of any secrets that it might obtain from a vault, and it does it the same way no matter the developer, technology, or platform. And, it knows what to do when a secret has been changed, so that the client users and code never have to respond to changing secrets or even know that they are changing.

About

Reduce Attack Surface

Any secret acquired and used increases your attack surface. Secretless acts as the mediator, removing direct communication between the client and the resource and shrinking your attack surface.

Worry-Free Secret Management

Increase developer productivity by allowing them to focus more on doing what they do best and less on security training and secret handling.

Transparent Secret Rotation

Rotation and safe secret handling is the responsibility of the ”secretless” client and thus, does not require any configurational changes.

Open Source and audited for security
